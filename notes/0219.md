## 자바 메모리 구조와 Static의 본질적 차이
### 1. 메모리 할당의 단위

- 내 질문
  - 패키지 단위로 메모리가 할당되는가? (X)

- 사실은? 
  - 자바의 메모리 로딩은 철저히 **'클래스 단위'**다. 같은 패키지 내 import 생략은 컴파일러의 편의 기능일 뿐, 메모리 구조와는 무관하다.

### 2. 메소드 영역(Method Area)

- 공통점
  - static 메소드든 instance 메소드든, 코드 자체(설계도)는 모두 메소드 영역에 단 한 번 로드되어 공유된다.

- 차이점:
  - Static: 객체 생성 여부와 상관없이 언제든 호출 가능 (주소만 알면 됨).
  - Instance: 반드시 **Heap 영역의 객체(인스턴스)**가 생성되어야만 호출 가능.

### 3. 어차피 둘 다 메모리에 있는데, 왜 인스턴스 메소드는 바로 못 부르나?

- 기술적 이유: 인스턴스 메소드는 실행 시 숨겨진 파라미터로 **this(객체의 주소/상태)**를 요구한다.

- 설계적 이유: '데이터(인스턴스 변수)' 없는 '로직(메소드)' 실행은 의미가 없거나 위험하다. (마치 재료 없이 요리법만 가지고 요리하려는 것과 같음)

- 또한, static으로 고정하면 자바의 핵심인 **다형성(Overriding)**을 사용할 수 없다.

### 4. 결론

- 메소드 내부에서 인스턴스 변수나 this를 전혀 사용하지 않는다면?

  → static으로 선언하는 것이 옳다.

이는 "이 메소드는 객체의 상태와 무관한 독립적인 도구(Utility)다"라는 설계 의도를 명확히 하고, 호출 성능을 미세하게 높여준다.
